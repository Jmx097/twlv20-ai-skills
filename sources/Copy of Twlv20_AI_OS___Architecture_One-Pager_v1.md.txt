Twlv20 AI OS — Architecture, Phase 1 Proposal & Agreement

Prepared by: Plinko Solutions (Jonathan McLemore) Prepared for: Jessie Hales, Twlv20 Version: 6.0 — merged with contract terms, liability, and acceptance Date: April 22, 2026

1. What we're building

An AI Operating System for Twlv20 — a multi-channel command layer plus an agent orchestration platform that lets Jessie (and a small core team) delegate real work to AI from Slack, WhatsApp, or voice, see everything happening through a single dashboard (Paperclip), and keep humans in the loop on anything that touches a client account or production system.

Phase 1 is a revenue-first sprint, not a generic build-out. The goal is to get Jessie out of the weeds on the work currently sitting on his plate (Pure Peptide Solutions, AgereSciences, Sales Recruiting University, plus internal agency work) and into a position where AI agents do the heavy lift while a small team of humans (Keeper, Njigo, Faguni) act as QA gates between steps.

Design principles

Conversational command surface. Jessie commands work from Slack, WhatsApp, or voice. The dashboard is for visibility and approvals, not for issuing tasks.

Human-in-the-loop by default, kill-switch always. Publishing a site, sending email, spending ad budget, modifying a client account — always gated by a named human (Keeper, Njigo, Faguni, or Jessie). Every agent has a manual-pause kill-switch. Per-sender/per-label Autopilot mode with an Undo window is available later, only after a workflow has earned the trust.

Compartmentalized client context. A single shared "brain" with per-client knowledge bases, secrets, tools, and SOPs. Agency-wide best practices live in a global layer; client-specific data never crosses tenants.

Open-source first / self-hostable. No vendor lock-in on the core stack.

Cost transparent. Every agent run is attributable to a tenant and a workflow via the shared Run-Log.

2. Division of labor

The system Plinko is delivering uses a ClickUp-based HITL pattern: agents do the orchestration and spec work, named humans execute. Phase 1 runs exactly the same way. Plinko is the orchestrator — securing accounts, writing the specs, running the workflows. Keeper is the execution hand on the infra side, picking up well-specified ClickUp tasks and shipping them.

That's not a workaround. It's the proof-point: by the time Phase 1 wraps, Keeper has already operated inside the same pattern his clients will.

            ├── Knowledge base (client-specific docs, SOPs, brand)

            ├── Agents (instantiated per tenant)

            ├── Runs (Run-Log — every agent execution)

            ├── Approvals (HITL queue, mirrored into Slack + ClickUp)

            └── Artifacts (drafts, reports, generated assets)

Row-level security on every table keyed by tenant_id. Jessie sees one pane across all tenants; per-tenant views available when he wants to drill in. Plinko writes the schema + RLS policy spec; Keeper ships the migrations.

Run-Log schema (shared across every agent and workflow): run_id, tenant_id, workflow, agent, timestamp, trigger_source, input_hash, confidence_score, HIL_status (pending / approved / rejected / auto), outcome, token_cost, api_cost, duration_ms, artifact_refs, error_payload. This is how cost attribution, audit trails, and performance monitoring all work — one table, one source of truth.

7. Human-in-the-loop model

Every workflow that touches a live system goes through a named human before it ships. The hand-offs are routed through ClickUp tasks so there's a paper trail. Every agent also ships with a manual-pause kill-switch surfaced in Slack and the Paperclip dashboard — one click halts the workflow.

Default chain for a website build:

AI agent drafts content (copy, structure) → Jessie or client review

AI agent generates design → Faguni review/tweak

AI agent builds the site (or hands HTML/CSS spec) → Keeper builds/QA

Agent sets up the GHL workflow tied to the site → Keeper QA's the GHL connection

Site goes live

Default chain for an analytics/health-check workflow (e.g., GHL audit, Facebook weekly report):

Agent runs read-only audit → produces report and recommendations

Report posted to Slack + filed in ClickUp

Jessie reviews; approved changes get queued as new agent or human tasks

Autopilot (optional, earned). Once a given sender / label / workflow has a consistent track record (high approval rate, low edit rate over N runs), Jessie can elect to put that narrow slice on Autopilot with an Undo window — the agent still writes to the Run-Log, and Jessie can revert inside the window. Default in Phase 1 is full HIL on everything.

The ClickUp side is itself a Phase 1 task — agents need an organized board to write into. See §9.

9. ClickUp as the work-handoff layer

ClickUp plays three roles in Phase 1 — the third being the build itself:

Phase 1 cleanup target. Jessie's current ClickUp has a lot of stale work in it. The first agent run on the live system is a cleanup + restructure pass that establishes the SOPs each of the three clients will operate under going forward.

The hand-off bus between agents and humans, once live. When an agent completes a step that needs a human (Keeper, Njigo, Faguni) or hits an approval gate, it writes a ClickUp task with full context (links, artifacts, what was done, what's needed). When the human completes their step, the next agent picks it up.

The hand-off bus between Plinko and Keeper during the build. The infra task package in §8 lives in ClickUp exactly as a post-go-live HITL queue would. By wrap-up, Keeper has already shipped 10+ tasks on the system's own operating model.

Phase 2 extends this so that as new work comes in, agents grounded in the knowledge base know which SOP applies, who the right human is, and what the chain of tasks should look like — without Jessie having to scaffold each project manually.

11. Knowledge base — how we seed it

Two parallel passes during Phase 1:

Jessie's existing materials. Jessie has a meaningful library of Claude skills, repos, and templates already saved — these get loaded into the global + marketing skill folders. (Action item: Jessie shares those out at or before Sprint 1.)

Google Drive reorganization. Jessie's Drive (personal + per client) needs to be reorganized into the folder structure the KB pulls from. This is itself a candidate first job for the agent — agents reorganize Drive against an SOP, Jessie reviews. Then Jessie spends focused time loading docs into the KB folders so the agents have what they need.

Per-client KB content includes: brand voice, offer/positioning, current campaigns, GHL pipeline structure, team-member responsibilities, process-flow definitions for how a task moves through the system.

13. The bigger picture

This Phase 1 build deliberately stays narrow: get Jessie's current backlog moving, prove the core architecture, get the team working alongside agents. Once it's running, the same architecture supports everything downstream — the custom Twlv20 dashboard with per-client tabs, multi-tenant isolation at scale, full Facebook Ads automation, the agency-launch idea Jessie and Jonathan talked about (build → market → scale → exit). Each becomes its own SOW, slotted into the same stack with no replatforming.

The pattern the build itself uses — Plinko orchestrates, specs live in ClickUp, Keeper executes — is the same pattern Twlv20's AI OS will run on every day after go-live. Phase 1 is the system proving itself on its own build.
